# RAG Memory System - 私有化 AI 记忆系统

> 给 AI 装一个「外脑」，让它记住你的一切

---

## 这是什么？

你有没有想过：如果 AI 能记住你看过的所有文章、写过的所有笔记、聊过的所有对话，会是什么体验？

这套系统就是干这个的。它把你的知识库变成一个「可检索的记忆库」，当你问 AI 问题时，它会先去你的记忆库里翻一翻，找到相关的内容，然后基于这些「记忆」来回答你。

**简单说：RAG = 检索 + 生成**。AI 先检索你的资料，再生成回答。

---

## 它能干什么？

- **帮你找东西**：「我上次写的那个技术方案在哪？」
- **帮你回忆**：「我和某朋友上次聊了什么？」
- **帮你回答**：「根据我的笔记，这个项目的主要风险点是什么？」

---

## 怎么工作的？

用一个类比来解释：

想象你有一个图书管理员（AI），但这个管理员是新来的，对你的藏书一无所知。每次你问问题，它都要从头翻一遍书架——太慢了。

于是你做了一个「索引卡片系统」：
1. 把每本书的内容切成小段
2. 给每个小段贴上「语义标签」（用向量表示）
3. 存进一个「卡片柜」（向量数据库）

现在你问问题的时候：
1. 管理员先把你的问题也贴上「语义标签」
2. 去卡片柜里找标签最相近的那些卡片
3. 把卡片内容读一遍，然后回答你

这就是 RAG 的基本原理。

## 系统架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        你（提问）                                │
│                          ↓                                      │
│                   代理网关 (8080端口)                            │
│                    ↓ 拦截请求，注入记忆                          │
│                          ↓                                      │
│                   海马体服务 (8000端口)                          │
│                    ↓ 混合检索                                    │
│         ┌────────────────┼────────────────┐                    │
│         ↓                ↓                ↓                    │
│    向量检索          BM25检索         Reranker精排              │
│    (语义相似)        (关键词匹配)      (深度打分)                │
│         └────────────────┼────────────────┘                    │
│                          ↓                                      │
│                   记忆切片 → 注入到 Prompt                       │
│                          ↓                                      │
│                   大模型 (Claude/GLM/...)                       │
│                          ↓                                      │
│                        回答你                                   │
└─────────────────────────────────────────────────────────────────┘
```

**核心组件：**

| 组件 | 作用 | 端口 |
|------|------|------|
| 代理网关 | 拦截请求，注入记忆，转发给大模型 | 8080 |
| 海马体服务 | 混合检索，从记忆库捞东西 | 8000 |
| 向量数据库 | 存储向量化后的记忆切片 | 本地文件 |
| BM25索引 | 关键词检索，弥补向量检索的不足 | 本地文件 |
| Reranker | 精排，从候选里挑最好的 | 内存 |
| 向量化引擎 | 把文本变成向量 | Ollama |

## 为什么要「混合检索」？

这是我踩过的最大的坑。

一开始我只用了向量检索，效果还行，但有个致命问题：**它对数字、日期、人名「脸盲」**。

比如你问「1月14号我干了什么」，向量检索可能会把 1月7号、1月21号的内容都捞出来——因为在语义空间里，它们都是「一月份的某一天」，太像了。

后来我加上了 BM25 关键词检索。BM25 就像疯狗一样做字面匹配，你搜「2024-01-14」，它就去所有文档里找这几个字，管你什么语义不语义。这样就解决了「脸盲」问题。

但两路检索结果怎么合并？用 RRF（倒数排名融合）算法。简单说就是：如果一条文档在两路检索里都排得很靠前，那它大概率是你想要的。

最后再加一个 Reranker 做精排。Reranker 会把你的问题和每条候选文档放在一起「深度阅读」，计算一个相关性分数。这一步比向量检索更精准，但计算量更大，所以只对候选集做，不做全量。

---

## 权重系统是什么？

另一个踩过的坑：噪音问题。

我的知识库里有大量微信聊天记录，全是「哈哈哈」「在吗」「表情包」。如果这些内容和核心笔记一起检索，很容易「喧宾夺主」。

一开始我的思路是「数据清洗」，把聊天记录全部过滤掉。结果有一天我问「某朋友是谁」，系统告诉我「未找到相关信息」——因为我把聊天记录过滤掉了，而人名、关系恰恰藏在那些「废话」里。

后来我改成「数据湖」思路：**不删任何数据，但给不同来源打上不同权重**。

- 「核心文档」目录 → 权重 1.0（最高优先级）
- 「工作笔记」目录 → 权重 0.7
- 「微信聊天」目录 → 权重 0.3（保底，但不会被过滤）

检索的时候，高权重的内容会优先浮上来。这样既保留了所有数据，又不会被噪音淹没。

## Prompt 设计

这是最容易被忽略、但最关键的一环。

我一开始用的 Prompt 是这样的：

```
请基于以下参考资料回答。如果参考资料中没有，请回答不知道。
```

结果 AI 变成了「复读机」，只会机械地复述检索结果，完全丧失了推理能力。你问它「老明是谁」，它会翻开聊天记录告诉你「老明的微信号是某某」，但它不敢推理「老明就是你本人」。

后来我把 Prompt 改成了更自然的方式：

```
你是我的赛博外脑。以下是你唤起的记忆切片，就把它们当做你脑子里自然浮现的信息。
该推理推理，该回答回答，别老是把「根据知识库」挂在嘴边。
```

这一改，AI 瞬间「活」过来了。它不再把记忆当成外在的参考资料，而是当成了自己潜意识的一部分。

**核心教训：别用 Prompt 把 AI 绑成只会念档案的傻子。**

## 安装步骤

### 1. 环境要求

- Python 3.11+
- Ollama（用于本地向量模型）
- 内存 8GB+（推荐 16GB）

### 2. 安装依赖

```bash
# 克隆仓库
git clone https://github.com/sgsss998/RAG_Memory_System.git
cd RAG_Memory_System

# 创建虚拟环境
python3 -m venv venv
source venv/bin/activate

# 安装依赖
pip install -r requirements.txt

# 安装 Ollama
# macOS: brew install ollama
# Linux: curl -fsSL https://ollama.ai/install.sh | sh

# 下载向量模型（约 2GB）
ollama pull bge-m3
```

### 3. 配置

```bash
# 复制配置文件
cp repos_config.json.example repos_config.json
cp weight_rules.json.example weight_rules.json

# 编辑 repos_config.json，填入你的知识库路径
nano repos_config.json

# 编辑 proxy_gateway.py，配置：
# 1. BASE_IDENTITY（你的身份锚点）
# 2. REAL_API_BASE（你的 LLM API 地址）
nano proxy_gateway.py
```

### 4. 构建索引

```bash
# 全量索引（首次运行）
python git_memory_sync.py --full

# 构建 BM25 索引
python build_bm25_index.py
```

### 5. 启动服务

```bash
# 启动海马体服务
python serve_memory_v2.py &

# 启动代理网关
python proxy_gateway.py &

# 测试
curl http://localhost:8000/
curl http://localhost:8080/
```

### 6. 客户端配置

在你的客户端机器上配置环境变量：

```bash
export ANTHROPIC_BASE_URL="http://YOUR_SERVER_IP:8080"
export ANTHROPIC_AUTH_TOKEN="your_api_key"
export ANTHROPIC_MODEL="your_model_name"
```

## 常见问题

**Q: 为什么用 Git 同步？**
成熟稳定，有版本控制，用 Deploy Key 可以做到安全隔离。

**Q: 为什么选 bge-m3？**
中英文都支持，本地部署隐私安全，效果也不错。

**Q: 为什么需要 BM25？**
向量检索对数字、日期、人名「脸盲」，BM25 做字面匹配可以弥补这个缺陷。

**Q: 为什么需要权重系统？**
防止聊天记录等噪音淹没核心内容。

**Q: 检索不准怎么办？**
1. 检查权重配置是否合理
2. 确认 BM25 索引已构建
3. 尝试全量重建：`python git_memory_sync.py --full`

**Q: 内存不够怎么办？**
1. 减少候选条数（在代码里调 VECTOR_TOP_K 和 BM25_TOP_K）
2. 禁用 Reranker
3. 分批处理文档

## 局限性

说实话，这套系统**不是万能的**。

- 它能帮你从数据里捞东西，但**不会凭空创造知识**
- 如果你从来没在笔记里写过「我的网站是某某」，它怎么也答不出来
- 它的「理解」能力取决于你写的 Prompt 和喂给它的数据质量

把它当成一个**增强版搜索引擎**就好，别指望它变成真正「懂你」的数字分身。但用好了，确实能帮你省很多翻笔记的时间。
